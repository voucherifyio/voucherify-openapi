The new loyalty program model defines how customers earn, keep, and spend points. In the Dashboard, you manage this model in **Loyalty hub**.

<Warning>
Migration from existing loyalty campaigns is not currently automated.

Customers, points, rewards, and campaign configuration from existing loyalty campaigns cannot currently be moved automatically to new loyalty programs. Voucherify is investigating migration support, but customers should expect integration changes.
</Warning>

## What changed

The new loyalty program model separates program setup into reusable components.

In existing loyalty campaigns, most settings belong to one campaign. This makes each campaign more isolated. If you need a similar setup in another campaign, you often need to configure it again.

In the new model, you connect components inside a loyalty program. This helps you reuse configuration, manage more complex setups, and support programs with multiple point wallets.

## Program components

The main program components match the sections available in **Loyalty hub**.

<AccordionGroup>

<Accordion title="Point wallets">

Point wallets define how points are stored and managed.

A loyalty program can include multiple point wallets. You can use them to create different point balances or point currencies inside one program.

Point wallets control point balances, pending points, expiration, earning limits, spending limits, refunds, and pay with points.

Point wallets are planned to support different membership models: individual, household, or team. Household wallets will support private point pooling, while team wallets will support public group participation.

</Accordion>

<Accordion title="Tier structures">

Tier structures group loyalty members into levels.

A loyalty program can include one tier structure. Tier structures can control qualification rules, expiration, grace periods, downgrade behavior, and tier-specific behavior.

Tier mappings let you adjust selected program rules for customers in a specific tier.

</Accordion>

<Accordion title="Earning rules">

Earning rules define when customers receive points or rewards for their actions.

An earning rule is based on a trigger, optional conditions, and one or more effects.

Available trigger events:

- **Order paid**: triggers the earning rule after an order is successfully paid
- **Custom event**: triggers the earning rule when a selected custom event is sent to Voucherify
- **Segment entered**: triggers the earning rule when a customer enters a selected segment

Effects define what happens when the rule applies, for example adding points to a point wallet or granting a reward.

</Accordion>

<Accordion title="Benefits">

Benefits grant points, material benefits, or digital benefits to eligible loyalty members.

Customers do not need to redeem benefits manually.

</Accordion>

<Accordion title="Rewards">

Rewards are items customers can redeem through the loyalty program.

Reward types include digital rewards and material rewards. Digital rewards can issue discount coupons or gift cards, while material rewards represent physical products or SKUs.

Rewards use spendings to define when customers can redeem them, which point wallet is used, and how many points the reward costs.

</Accordion>

</AccordionGroup>

## How components work together

A loyalty program connects all components into one customer experience.

A simplified structure looks like this:

- the loyalty program contains the full setup
- point wallets store and manage points
- earning rules decide when customers earn points or rewards
- rewards define what customers can redeem
- tier structures group customers into levels
- benefits grant points, material benefits, or digital benefits automatically

This structure makes loyalty programs more flexible. Instead of building every rule directly inside one campaign, you create components and connect them where they are needed.

For example, one program can use multiple point wallets. One reward can have different prices in different wallets. One tier can change earning limits or reward prices for customers who qualify for it.

## How points move through a program

Points move through a loyalty program based on earning rules and point wallet settings.

A simple earning flow looks like this:

1. A customer performs an action.
2. An earning rule is triggered.
3. Conditions are checked.
4. The first matching earning configuration is applied.
5. The customer receives points or a reward.
6. Points are stored in the selected point wallet.
7. Wallet settings control pending points, expiration, limits, refunds, and spending.
8. The customer can redeem rewards, pay with points, qualify for tiers, or receive benefits.

This flow can stay simple for basic programs. It can also support more complex programs with multiple wallets, tier-specific mappings, and advanced point management.

## Point management

Point wallets define how points behave after they are earned.

They control when points become available, how long points remain valid, how many points customers can earn or spend, and what happens when an order is refunded.

For household and team wallets, earning and spending limits are tracked at the member level.

<AccordionGroup>

<Accordion title="Pending points">

Pending points define when earned points become available.

You can make points available immediately, after a defined period, on fixed activation dates, or after a selected activation event.

For manual pending point activation, the activation event must be sent within one year. If the event is not sent within one year, the pending points are removed.

</Accordion>

<Accordion title="Point expiration">

Point expiration defines how long points remain valid.

Points can have no expiration, expire after a rolling period, expire on calendar dates, or expire after a period of inactivity.

</Accordion>

<Accordion title="Earning limits">

Earning limits control how many points a customer can collect.

You can use earning limits to set global limits for a period of time or transaction-level limits for a single transaction.

</Accordion>

<Accordion title="Spending limits">

Spending limits control how many points a customer can use.

You can use spending limits to set global limits for a period of time or transaction-level limits for a single transaction.

</Accordion>

<Accordion title="Refunds">

Refunds define what happens to points when an order is refunded.

Refund settings can remove points earned from a refunded transaction, return points spent on a refunded transaction, or support partial refund handling.

</Accordion>

</AccordionGroup>

## Differences from existing loyalty campaigns

The new loyalty program model expands what marketers can build compared with existing loyalty campaigns.

Existing loyalty campaigns are more isolated. They use one point balance and keep most configuration inside one campaign. New loyalty programs use reusable components, multiple point wallets, wallet-specific reward prices, and more advanced tier behavior.

| Area | Existing loyalty campaigns | New loyalty programs |
| --- | --- | --- |
| Program structure | Most configuration belongs to one campaign | Configuration is separated into reusable components |
| Point wallets | One point balance per campaign | Multiple point wallets in one program |
| Earning logic | One earning configuration | Multiple earning configurations, with the first matching configuration applied |
| Rewards | One reward price | One reward can be connected to multiple point wallets with different prices |
| Tiers | One tier structure in one campaign | One tier structure in one program, with tier mappings for selected rules |
| Households and teams | Not supported | Household and team wallets are planned to support shared earning models |

This comparison focuses on the main model changes. Detailed configuration options are covered in the dedicated feature articles.

## Program limits

Program limits define how large one loyalty program can be.

Current limits:

- up to 10 point wallets in one loyalty program
- up to 100 rewards in one loyalty program
- up to 100 earning rules in one loyalty program
- one tier structure in one loyalty program
- up to 100 tier levels in one tier structure

These limits help you decide when to keep configuration in one program and when a separate program may be needed.

## Current status

The new loyalty program model is available as early access.

The current focus is modular program creation, reusable resources, multi-wallet setups, and basic tiering.

Existing loyalty campaigns remain available for customers who already use them. New loyalty features will be added only to the new loyalty program model. Existing loyalty campaigns will receive bug fixes only.

Migration is not automated at this time. Voucherify is investigating how to help customers move their data to the new model, but the new model uses a different customer-facing integration, so migration will require integration changes.

## Why use the new loyalty program model

The new model is useful when a program needs more flexibility than one point balance and one isolated campaign setup.

**Reusable configuration**

Create components once and connect them where they are needed.

**Multiple point wallets**

Manage different point balances or point currencies in one program.

**More control over points**

Configure pending periods, expiration, earning limits, spending limits, refunds, and pay with points.

**Tier-specific behavior**

Use tier mappings to adjust selected rules for different customer levels.

**Larger program structures**

Build complex loyalty setups without creating separate isolated campaigns for every variation.
